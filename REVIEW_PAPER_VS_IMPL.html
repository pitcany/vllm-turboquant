Known Issues After Trim

  • No-alloc decode fallback returns zeros when TQ store has <16 tokens
  • No-alloc prefill uses repeat_interleave for GQA, massive memory at 200k
  • Value group_size hardcoded to 32 (now explicit, was previously fragile math)
  • MLA path is passthrough — no quantization for DeepSeek V3 etc.
  • 200k baseline stalls — no dual-case telemetry available
  • Quality benchmarks are synthetic — no LongBench/RULER runs

Quantization Pipeline

graph LR
    subgraph "Key: TurboQuant_Prod"
        K["Key (H, T, D)"] --> NORM["||x||, normalize"]
        NORM --> ROT["y = x @ Pi_T"]
        ROT --> IDX["searchsorted -> indices"]
        IDX --> PACK["bit-pack (b-1 bits)"]
        ROT --> DQ["dequant MSE"]
        DQ --> RES["r = x - x_mse"]
        RES --> QJL["sign(r @ S_T)"]
        QJL --> SIGNS["pack signs (1 bit/coord)"]
        RES --> RNORM["||r||"]
    end

        GRP --> MM["per-group min/max"]
        MM --> VQ["round + bit-pack (2-bit)"]
    end

Summary

What Works

  • Core TQ_MSE + TQ_Prod faithful to paper
  • All 3 theorems validated empirically
  • Needle retrieval passes at 8192 tokens
  • 5.1x compression vs FP16
  • 30k A/B: +5.7% prefill, +3.1% output speed
  • 200k TQ completed (199,952 tokens)

What Doesn't

  • MLA: stub only (DeepSeek V3 unsupported)
  • FlashInfer: incompatible
  • 200k quality unvalidated (no LongBench)
  • Multi-sequence batching untested
  • No-alloc prefill slow (SDPA, not flash)
Key gap: The "identical output" claim holds only for trivial prompts. Long-context retrieval quality is unquantified against standard benchmarks (LongBench, RULER, Needle-in-Haystack at 200k). The 200k needle failure in the handoff doc is concerning.
